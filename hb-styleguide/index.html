Overview

The styleguide is a reference of UI patterns used throughout the Homebase app.

It attempts to ensure that we're writing CSS in a manageable way, that we're providing a consistent experience to our users, and that our designer never has to design a popup again.

This is an alpha version and should serve as a conversation-starter for us to move towards a better front-end. Feedback is welcome and strongly encouraged.

Let's get our shift together!

CSS Guidelines

Formatting

  • Use soft tab (2 spaces) for indentation.
  • Never use ID selectors.
  • When using multiple selectors in a rule, give each selector its own line.
  • Put a space before the opening brace { in rule declarations
  • In properties, put a space after, but not before, the : character.
  • Put closing braces } of rule declarations on a new line
  • Put blank lines between rule declarations
Bad
.avatar{
    border-radius:50%;
    border:2px solid white; }
.no, .nope, .not_good {
    // ...
}
#lol-no {
  // ...
}
Good
.avatar {
  border-radius: 50%;
  border: 2px solid white;
}

.one,
.selector,
.per-line {
  // ...
}

Javascript Hooks

Avoid binding to the same class in both JS and CSS.

We recommend prefixing all JS hooks with .js-:

<button class="btn btn-primary js-request-to-book">Request to Book</button>

OOCSS & BEM

OOCSS & BEM are used throughout this styleguide and recommended for future use.

OOCSS (Object Oriented CSS) in a nutshell:

Separate skin and structure. Instead of adding a gradient to a button, create a global .gradient {...} class that can be reused elsewhere.

Separate container and content. An object should look the same wherever you put it. So instead of styling a specific h2 with .myObject h2 {...}, create a class that describes the h2 in question, like <h2 class="category-title">.

More info from people smarter than me:

  • https://github.com/stubbornella/oocss/wiki
  • https://css-tricks.com/bem-101/
  • http://csswizardry.com/2013/01/mindbemding-getting-your-head-round-bem-syntax/

BEM (Block, Element, Modifier) in a nutshell:

BEM is a naming convention for HTML & CSS classes. Here's an example of what BEM looks like:

/* Block component */
.btn {}

/* Element that depends upon the block (two underscores) */ 
.btn__price {}

/* Modifier that changes the style of the block (two dashes) */
.btn--orange {} 
.btn--big {}

The markup would look like this:

<a class="btn btn--big btn--orange" href="http://css-tricks.com">
  <span class="btn__price">$9.99</span>
  <span class="btn__text">Subscribe</span>
</a>
"God, that HTML looks awful".

At first, yes, BEM looks awful. However, using it means that:

  • Any developer can read markup and understand the relationships between HTML & CSS. For example, if we were using DRY CSS we might write <div class="box profile pro-user">. How are the classes box and profile related to each other? Are they related at all? Can you use .pro-user anywhere else? Using the BEM naming convention changes all of that.
  • Everything is reusable. Components can be place anywhere and look exactly the same, as they have no dependency on pages or other DOM elements. This means that nested selectors are strongly discouraged - nesting increases coupling and makes code reuse impossible.
  • The amount of new CSS we have to write is dramatically reduced.
